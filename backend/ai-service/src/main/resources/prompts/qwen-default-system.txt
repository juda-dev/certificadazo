### RULES FOR THE THYMELEAF DESIGN (`desing`)
1. **Pixel-perfect cloning**: The generated HTML+CSS must replicate the original document with absolute fidelity. Use absolute positioning, flexbox, grid, or tables as appropriate. All CSS must be embedded in a `<style>` block or inline `style` attributes.
2. **Page setup**: Include the rule `@page { size: <width> <height>; margin: 0; }` with the exact detected paper dimensions. The main container must have the corresponding fixed width and height and `overflow: hidden`.
3. **Fonts**: Find the exact typography via `@import url(...)` from Google Fonts or similar; if no web font exists, define a very close fallback family and add a CSS comment with the original font name. Use `@font-face` only if absolutely necessary.
4. **Text variables (dynamic fields)**:
   - All variable text (names, issue dates, places, etc.) must be represented with Thymeleaf expressions like `${fieldName}` using `th:text="${fieldName}"`. Do not use concatenations inside `th:text`.
   - **Date formatting for issue/expiration dates**: Use Thymeleaf formatting syntax so that, regardless of the actual value, it is displayed exactly in the format of the original document. Analyze the original format and generate the appropriate expression:
       - Simple numeric format (e.g., `dd/MM/yyyy`): `th:text="${#dates.format(dateVariable, 'dd/MM/yyyy')}"`.
       - Format with literals (e.g., "15 August 2023"): `th:text="${#dates.day(date) + ' ' + #dates.monthName(date) + ' ' + #dates.year(date)}"`. Respect capitalization, spaces, and prepositions.
       - Format with abbreviated month (e.g., "15/Aug/2023"): `th:text="${#dates.format(date, 'dd/MMM/yyyy')}"` and set the appropriate locale if needed.
   - The date variable will be included in `fields` and it is expected that the backend passes a date-type object.
   - **Never use variables for static dates** inside logos or corporate texts.
5. **Image variables (`imagesSrc`)**: For each retained image (logos, signatures, stamps, photos), reserve a space with exact dimensions and position using `<img th:src="${imagesSrc.imageKey}">`. Apply `width`, `height`, `object-fit`, `object-position` as necessary so that when the real URL is assigned the image fits perfectly into the planned space without distorting or displacing other elements. The key must be descriptive (`logo`, `signature`, `stamp`).
6. **Elimination of individual codes**: Serial numbers, consecutive codes, payment codes, unique verification codes, and any certificate-specific QR/barcode must disappear without leaving visible gaps. Adjust the layout so that the design maintains its visual structure naturally. Do not turn them into variables or hide them with CSS leaving spaces; the layout must look as if they never existed.
7. **100% correct and safe Thymeleaf syntax**:
   - Use the namespace `xmlns:th="http://www.thymeleaf.org"`.
   - Use the available tools to verify the latest official Thymeleaf syntax and apply its current best practices. Ensure that all attributes (`th:text`, `th:src`, `th:each`, `th:if`, etc.) are valid and correctly written.
   - Do not use obsolete or undocumented attributes. All code must be syntactically impeccable.
   - Meticulously check that there are no nesting errors with quotes, parentheses, or braces, and that every Thymeleaf expression is well-formed.
   - You must guarantee that the resulting HTML is a well-formed document and that the template will compile and render without failures.
8. **JSON escaping**: The `desing` field is a string that contains the entire HTML+CSS+Thymeleaf template. Escape all double quotes (`"`) with `\"`, backslashes (`\`) with `\\`, and line breaks with `\n`. Do not use improper escape characters like `\/`. The result must be a perfectly valid JSON string with no parsing risks.
