### RULES FOR THE THYMELEAF DESIGN (`desing`)
1. **Pixel-perfect cloning**: The generated HTML+CSS must replicate the original document with absolute fidelity. Use absolute positioning, flexbox, grid, or tables as appropriate. All CSS must be embedded in a `<style>` block or inline `style` attributes.
2. **Page setup**: Include the rule `@page { size: <width> <height>; margin: 0; }` with the exact detected paper dimensions. The main container must have the corresponding fixed width and height and `overflow: hidden`.
3. **Fonts**: Find the exact typography via `@import url(...)` from Google Fonts or similar; if no web font exists, define a very close fallback family and add a CSS comment with the original font name. Use `@font-face` only if absolutely necessary.
4. **Text variables (dynamic fields)**:
   - All variable text (names, issue dates, places, etc.) must be represented with Thymeleaf expressions like `${fieldName}` using `th:text="${fieldName}"`. Do not use concatenations inside `th:text`.
   - **Date formatting for issue/expiration dates**: Use Thymeleaf formatting syntax so that, regardless of the actual value, it is displayed exactly in the format of the original document. Analyze the original format and generate the appropriate expression:
       - Simple numeric format (e.g., `dd/MM/yyyy`): `th:text="${#dates.format(dateVariable, 'dd/MM/yyyy')}"`.
       - Format with literals (e.g., "15 August 2023"): `th:text="${#dates.day(date) + ' ' + #dates.monthName(date) + ' ' + #dates.year(date)}"`. Respect capitalization, spaces, and prepositions.
       - Format with abbreviated month (e.g., "15/Aug/2023"): `th:text="${#dates.format(date, 'dd/MMM/yyyy')}"` and set the appropriate locale if needed.
   - The date variable will be included in `fields` and it is expected that the backend passes a date-type object.
   - **Never use variables for static dates** inside logos or corporate texts.

  #### Special handling of contextual dates
When analyzing the document, you must perform an exhaustive contextual analysis of every detected date before deciding how to reproduce it.

Examples of contextual reasoning include (but are not limited to):

- An expiration date must always be after its corresponding issue date.
- A completion date must always occur before the certificate issuance.
- A payment date should be coherent with the transaction context.
- Any chronological relationship present in the original document must be preserved.

- Never generate impossible, contradictory or illogical timelines.

- If removing a contextual date produces a sentence that sounds incomplete or unnatural, rewrite only the affected sentence so that it remains perfectly natural while preserving the original meaning, visual layout and grammatical structure.

- If a contextual date can simply be omitted without affecting readability or the document structure, omit it completely instead of creating unnecessary placeholders or variables.

- Do not create additional fields inside `fields` for these contextual dates.

- Do not replace these contextual dates with `${...}` expressions.

- Historical dates that are part of the actual content (for example historical events, laws, constitutions, establishment years, commemorative dates or fixed corporate information) must remain exactly as they appear in the original document.

- Dates embedded inside logos, seals, signatures, photographs or any graphical asset must never be recreated as HTML text. They belong to the corresponding image and must remain inside that image.

- Before generating the final template, verify that every generated date is chronologically coherent with every other date appearing in the document and with the current date used as temporal reference.

5. **Image variables (`imagesSrc`)**: For each retained image (logos, signatures, stamps, photos), reserve a space with exact dimensions and position using `<img th:src="${imageKey}">`, where `imageKey` is the exact descriptive key for that image (e.g. `${logo}`, `${signature}`), referenced directly as a top-level variable and never nested under an `imagesSrc` object.
